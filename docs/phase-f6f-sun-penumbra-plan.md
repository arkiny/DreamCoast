# F6F 계획서 — 각반경-정합 태양 페넘브라: gv_shadow의 콘 기울기를 레퍼런스 태양에서 유도

> 상태: **진행 중 — 계획** (2026-07-16). 동기 =
> [phase-f6e-scatter-tolerant-gate-plan.md](phase-f6e-scatter-tolerant-gate-plan.md) §4c —
> hardvis(교차-전용)는 블록 게이트에서도 sunlit 과주입으로 기각(20.4 > 19.1)이고, 레거시
> k=16은 E-도메인 태양-체인 결손(lit-E 36.4 vs 레퍼런스 48.8, coarse 시대 측정)의 유력
> 상류다. 목표 = 물리적으로 정의된 중간값 랜딩과 hardvis+fine 콤보가 보였던 interior
> 25.735급 회수.

## 0. 단일소스 사실 (정정 포함)

- **레퍼런스의 태양은 유한 디스크다**: `SUN_COS_MAX = 0.9998`(rt_common.slang — NEE가
  `sample_cone`으로 샘플) = 각반경 **0.02 rad(1.146°)**.
- 구면-추적 소프트 섀도의 콘 기울기 관계: penumbra 계수 = `k·h/t`에서 k = cot(각반경).
  레퍼런스-정합 값 = **cot(0.02) ≈ 50**. 레거시 k=16(3.58°)은 레퍼런스 태양의 **×3.1
  과대**(과암화 — 5m 거리에서 10cm 근접-통과가 68% 차폐).
- **기록 정정**: E-오라클 §2d와 F6E의 "태양각반경×13"은 실제 태양(0.265°) 기준 서술이었다.
  패리티 게이트의 기준은 레퍼런스 자신의 태양(1.146°)이므로 본 페이즈의 정합 목표는 k≈50이다
  (hardvis ≈ k→∞는 그보다도 4× 이상 하드 — sunlit 과주입의 기전).

## 1. 설계

- `SUN_COS_MAX`를 **sky_common.slang로 이동**(rt_common·gi_volume 모두 sky_common을
  인클루드 — 단일 정의) + 파생 상수 `SUN_PENUMBRA_K = SUN_COS_MAX / sqrt(1 −
  SUN_COS_MAX²)` (= cot θ ≈ 50.0).
- gi_volume push **196→200B**: `sun_k`(f32) — **0 = 레퍼런스-유도 상수 사용**(권위 값은
  셰이더의 SUN_COS_MAX에 남음 — 호스트 중복 금지), >0 = 명시 오버라이드(스윕용).
  호스트 기본 = 16.0(레거시, 바이트-불활성 커밋). env `P_GI_SUN_K`.
- 기존 hardvis(flags bit1)는 유지(k 스윕의 ∞ 끝점·진단용); k 랜딩 시 주석으로 관계 명시.

## 2b. 사후 부검 — fine 랜딩의 스토리지 지연 버그 (정오표, 2026-07-16)

k-스윕 착수 시 k=16 행이 **coarse-시대 VP-D 캡처와 바이트-동일**함을 발견 → 추적 결과
`P_GI_VOL_CLIP`의 **3개 판독 사이트 중 볼륨 생성 사이트(gi.rs — fine 하프의 높이 2×
할당)만 기본 false로 남아** 있었다. 즉 F6E의 "fine 기본 ON" 커밋(9d9e9ea)은 실제로는
**coarse 스토리지 + fine-조건부 tint 0.15**를 봉인했다(당시 sunlit bias ≈0 붕괴는 tint
효과). 수동 쌍(P_GI_VOL_CLIP=1 명시)과 공식 러너의 소차(19.142 vs 18.799)가 이 버그의
증거였는데 러너-우선 규약으로 오독했다.

수습: ① 생성 사이트 기본 true(3사이트 락스텝 — 동일 knob의 모든 기본 판독은 일치해야
한다는 클래스 교훈, 전수 감사로 타 knob 무병 확인) ② **오봉인 예산 정오표**: sunlit
19.1은 오표기 구성의 측정치였으므로 직전 유효 봉인 19.2로 복원(스크립트 편집 + --update
재생성, 커밋에 문서화) ③ 진짜 fine 재봉인: sunlit **19.142 ≤ 19.2**(bias −0.03), interior
**26.996 ≤ 27.3(27.47에서 추가 하향!)**, sc_viz `dd97d557`(2런 결정론·시각 리뷰), 셔머
0.0503. coarse+t0.15(어제의 우연 구성)은 sunlit 18.799로 더 낮지만 **색 캘리브레이션
혈통이 없고**(t0.15는 F4B가 fine 스택용으로 앵커 스윕; coarse의 색 앵커는 0.2) 구조
원칙(루트-코즈 우선)에 따라 fine+t0.15를 기본으로 확정. 색 앵커 섀도: interior Δ(B−R)
−1.12(온난 — bit0의 청색-수입 제거가 F4B 캘리브 이후 필드를 데운 것) — k 랜딩 후 tint
재캘리브에서 함께 처리(§2-3).

## 2. 증분

1. **심 커밋**: 상수 이동+파생, push/knob 배선, 기본 16.0 바이트-불활성 — 풀 매니페스트
   비트-동일 증명. (PT는 상수 이동만 — 값 동일, 재컴파일만.)
2. **E-도메인 스윕**(커밋 없음, 현행 기본 스택 = fine ON 위에서): k ∈ {16, 32, 50, 100}
   + hardvis(∞) — EV11 뷰15 interior lit-E vs 레퍼런스 E-뷰(48.8). 후보 = lit-E가
   레퍼런스에 접근하되 초과하지 않는 최소-변경 k(50이 사전 등록 본명).
3. **캘리브레이션 커밋**: 승자 k를 기본으로(0=유도 모드 권장), 공식 배터리(block64
   19.1/27.47 하향만·셔머·갤러리·sc_viz 재시드)·hardvis+fine 콤보와의 비교 기록.
   실패 시(양 게이트 창 공집합) knob 잔존+판정 기록으로 종료(F4B3 방식).

## 3. 게이트·비목표

게이트: block64 예산(하향만), B−R 추적(섀도), 셔머, 갤러리 바이트, 풀 매니페스트, 검증된
단일 커밋. 비목표: PT 본체 수정(SUN_COS_MAX 값 변경 금지 — 위치만 이동) / tint 동시 변경
(k 단독 효과 고립 후 필요 시 후속) / hardvis 재스윕(끝점 데이터 재사용).
