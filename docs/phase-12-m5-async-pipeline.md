# Phase 12 M5 — 비동기 파이프라인 컴파일 + DefaultLoading 폴백

> **상태:** 🧪 계획 (미구현). 런타임 렌더러 인프라 — [ROADMAP](ROADMAP.md) Phase 12.
> **연관:** [shader-asset-cache.md](shader-asset-cache.md)(M4 빌드타임 쿡 캐시, **별개 레이어**).
> **무관(별개 트랙):** 런타임 Slang 인프로세스 *소스* 컴파일·리플렉션·핫리로드([shader-system-todo]).

## 1. 문제 (Why)

용어 정리 — 이 엔진에서 "셰이더 컴파일"은 **두 단계**다.

1. **빌드타임 바이트코드 컴파일** (`build.rs`, slangc → SPIR-V/DXIL/metallib). M4 쿡 캐시가 담당.
   런타임에는 `load_shader_pair`가 `&'static [u8]`을 즉시 반환 — **런타임 컴파일 아님**.
2. **런타임 파이프라인(PSO) 생성** (`Device::create_graphics_pipeline` / `create_compute_pipeline`).
   드라이버가 바이트코드를 **백엔드 ISA로 컴파일**한다(D3D12 PSO, VkPipeline, MTLRenderPipelineState).
   이게 런타임에 비싸고 스톨을 유발하는 단계 — 게이머가 보는 "Compiling shaders…"가 이것.

현재 샌드박스는 **시작 시 ~22개 파이프라인을 메인 스레드에서 동기 생성**(9개 모듈)하고 이후 매 프레임
재사용한다. 결과:
- **시작 스톨:** 모든 PSO가 첫 프레임 전에 동기 컴파일 → 프레임 1 지연. (현 씬은 ~1s로 작지만,)
- **확장성:** 범용 게임은 런타임에 **머티리얼/레벨 스트리밍**으로 새 PSO를 계속 만든다 → 그때마다 hitch.
  지금 구조는 새 PSO를 만드는 동안 프레임을 **블로킹**한다.

목표(사용자 요청): **PSO가 준비될 때까지 해당 패스의 드로우를 지연**하고, **완료되면 그린다**. 비싼 PSO
생성을 프레임 경계 밖/배경으로 빼서 스톨을 숨긴다. 머스트-드로우 패스(최종 톤맵/프레젠트 등)는 항상 상주하는
**DefaultLoading** 파이프라인으로 폴백.

> **정직한 범위:** 현재 씬에서 시작 PSO 비용은 작다(~1s, 22 PSO). M5의 실가치는 **미래 게임의 런타임
> 스트리밍**(많은 머티리얼·레벨 인/아웃)이다. CLAUDE.md "미래 게임을 위한 재사용 production code" 원칙에 따라
> 지금 인프라를 깔아 둔다. 측정 기준은 "최종 프레임 무회귀 + 지연 프레임이 점진적으로 채워짐".

## 2. 설계 개요 (What)

**파이프라인 준비성(readiness)을 1급 개념으로.** 각 패스는 자신의 PSO 핸들을 **`PipelineSlot`**(준비됨/대기중)
으로 들고, 렌더 루프는 슬롯이 준비됐을 때만 그 패스를 인코딩한다.

```
PipelineSlot::Pending   → 생성 미완. 패스 드로우 지연(skip) 또는 DefaultLoading 폴백.
PipelineSlot::Ready(p)  → 정상 인코딩.
```

생성 주체는 단계적으로 강화한다(아래 마일스톤). 핵심 불변식:
- **최종(warm) 프레임은 현재와 byte/pixel 동일** — 모든 PSO 준비 후 렌더 결과가 회귀 없어야 한다(양 백엔드
  픽셀 일치, 검증 클린). M5는 *언제* 그리는지만 바꾸고 *무엇을* 그리는지는 불변.
- **지연은 시각적으로 안전** — 준비 안 된 패스를 건너뛰어도 파이프라인/배리어 상태가 깨지지 않아야 한다
  (G-buffer 같은 의존 패스가 빠지면 소비처도 같은 프레임에 함께 지연).

## 3. 스레드 안전성 (the crux)

RHI는 enum-dispatch 파사드(`Device::Vulkan|D3d12|Metal`)이고 `create_*_pipeline(&self)`는 동기다. 백엔드
**드라이버**는 파이프라인 생성이 free-threaded다(D3D12 `CreateGraphicsPipelineState`, VK `vkCreatePipelines`
+ thread-safe `VkPipelineCache`, Metal `newRenderPipelineStateWithDescriptor`). 그러나 **우리 Rust 래퍼가
`Send`/`Sync`인지는 미검증**(raw 포인터/핸들 다수 → 기본 `!Send`). 이게 배경 스레드 생성의 관문이다.

→ **단계적 접근으로 위험 격리:** M5.1은 **메인 스레드**에서 *증분/지연* 생성만 한다(스레드 안전성 불필요,
저위험). 배경 스레드(M5.2)는 백엔드별 `Send`/`Sync` 감사 후 별도로.

## 4. 마일스톤 (각 게이트: fmt + clippy(-D) + **warm 프레임 양 백엔드 픽셀 동일** + 검증 클린)

### M5.1 — 준비성 + 지연 드로우 (메인 스레드 증분 생성) — 저위험, 사용자 동작 직접 충족
- `PipelineSlot{Pending,Ready}` 도입. 시작 시 전부 동기 생성하지 않고 **프레임당 N개씩**(또는 최초 사용 시)
  lazy 생성 → 비용을 여러 프레임에 분산, 씬이 **점진적으로 등장**.
- 렌더 루프: 슬롯이 `Pending`인 패스는 이번 프레임 **드로우 지연(skip)**, 다음 프레임 재시도. 의존 체인은
  함께 지연(소비처가 생산자 슬롯을 확인).
- 스레드 없음 → `Send`/`Sync` 무관, 백엔드 변경 없음. 동작: "준비될 때까지 안 그리다가 완료되면 그림"을
  그대로 구현(사용자 표현 일치).
- **검증:** 모든 PSO 준비 후 프레임이 현재와 픽셀 동일; 초기 프레임들은 부분 렌더(점진), 검증 클린; 헤드리스
  스크린샷은 "전부 준비될 때까지 워밍업 프레임"을 돌린 뒤 캡처(현행 캡처가 이미 몇 프레임 워밍업).

### M5.2 — 배경 스레드 컴파일 (선택·후속, 스레드 안전성 감사 후)
- PSO 생성을 워커 스레드로. 백엔드별 `Send`/`Sync` 감사 + 필요한 `unsafe impl`(드라이버 free-threaded 근거
  주석). 완료된 PSO를 메인 루프로 채널 전달 → 슬롯 `Pending→Ready`. 메인 프레임 시간에서 생성 비용 완전 제거.
- **VkPipelineCache**/D3D12 라이브러리 공유로 중복 컴파일 회피(선택).
- 위험: 드라이버 스레드 안전성/수명. M5.1 인프라 위에 얹으므로 폴백(메인 스레드 생성)은 항상 가능.

### M5.3 — DefaultLoading 폴백 (선택)
- 항상 임베드되는 **최소 셰이더/PSO**(예: 단색/체커 또는 직전 프레임). 머스트-드로우 패스(최종 프레젠트)가
  아직 `Pending`일 때 화면이 비지 않도록 상주 폴백으로 그린다. 빌드타임에 보장 임베드(쿡 캐시와 동일 경로,
  절대 `None` 불가) → "필수로 첨부되는 DefaultLoading 셰이더".

## 5. 검증 (전 마일스톤 공통)
- **warm 무회귀:** 모든 PSO 준비 후 `--screenshot-clean` 양 백엔드 픽셀 동일(DX≡VK ≤0.001), 검증 클린.
- **지연 정확성:** 초기 프레임에서 미준비 패스가 깨끗이 빠지고(배리어/레이아웃 오류 없음) 이후 채워짐.
- **무패닉/누수:** 슬롯 상태 전이에 레이스/누수 없음(M5.2는 채널·수명 점검).

## 6. 미결 / 결정 항목
- 증분 생성 속도(프레임당 N) 및 우선순위(가시 패스 우선?).
- 헤드리스 캡처의 "워밍업까지 대기" 정책(전부 Ready까지 N프레임 한도).
- M5.2 착수 여부(현 씬 이득 작음 → 스트리밍 시나리오 생기면). M5.1만으로 사용자 요청 동작은 충족.
- DefaultLoading 비주얼(단색 vs 직전 프레임 vs 로딩 패턴).
